## 2.5 CREATE TABLE ile Şema Tasarımı

Önceki başlıklarda:

- Veri tipleri,  
- Constraint’ler,  
- Primary key ve foreign key kavramları  

ayrı ayrı ele alındı. Bu bölümde, bu kavramları bir araya getirerek **tam bir tablo tanımının (DDL – Data Definition Language)** nasıl yazıldığını göreceğiz.

---

### 2.5.1 DDL ve CREATE TABLE Nedir?

SQL komutları genel olarak iki ana grupta ele alınabilir:

- Veri tanımlama komutları (**DDL – Data Definition Language**)  
  - Örnek: `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`  
- Veri işleme komutları (**DML – Data Manipulation Language**)  
  - Örnek: `SELECT`, `INSERT`, `UPDATE`, `DELETE`  

Bu bölümde DDL tarafında, özellikle `CREATE TABLE` komutuna odaklanıyoruz.

`CREATE TABLE`, yeni bir tabloyu, sütunlarını ve sütunlara ait kuralları (veri tipi, constraint vb.) tanımlamak için kullanılır.

---

### 2.5.2 Basit Bir Örnek: Students Tablosu

Aşağıda, temel bir `students` tablosunun nasıl tanımlanabileceğine ilişkin örnek yer almaktadır:

```sql
CREATE TABLE students (
    id          INTEGER PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    age         INTEGER CHECK (age >= 0),
    department  VARCHAR(100),
    gpa         NUMERIC(3, 2)
);
```

Bu tanımda:

- `id` sütunu, tablonun primary key alanıdır.  
- `name` sütunu boş geçilemez (`NOT NULL`).  
- `age` için negatif değer girilmesini engelleyen bir `CHECK` constraint tanımlanmıştır.  
- `department` serbest bırakılmış, zorunlu olmayan bir alandır.  
- `gpa`, virgüllü sayı tutmak üzere tanımlanmıştır.  

Tek bir `CREATE TABLE` ifadesiyle:

- Veri türlerini,  
- Zorunluluk kurallarını (NOT NULL),  
- Mantıksal kısıtları (CHECK),  
- Birincil anahtarı (PRIMARY KEY)  

bir arada tanımlamış olduk.

---

### 2.5.3 İlişkili Tablolar: Users ve Orders

Şimdi, iki tablo arasında ilişki kuran bir örnek inceleyelim. Basit bir e-ticaret senaryosu düşünelim:

- `users` tablosu: kullanıcı bilgilerini tutar.  
- `orders` tablosu: verilmiş siparişleri tutar.  

#### Users Tablosu

```sql
CREATE TABLE users (
    id        INTEGER PRIMARY KEY,
    name      VARCHAR(100) NOT NULL,
    email     VARCHAR(255) UNIQUE NOT NULL,
    is_active BOOLEAN DEFAULT TRUE
);
```

Açıklamalar:

- `id`: primary key olarak tanımlanmıştır.  
- `email`: hem `UNIQUE` hem de `NOT NULL` olacak şekilde tanımlanmıştır.  
- `is_active`: varsayılan değeri `TRUE` olan bir Boolean alandır.  

#### Orders Tablosu

```sql
CREATE TABLE orders (
    id          INTEGER PRIMARY KEY,
    user_id     INTEGER NOT NULL,
    amount      NUMERIC(10, 2) NOT NULL,
    created_at  TIMESTAMP NOT NULL,

    CONSTRAINT fk_orders_users
        FOREIGN KEY (user_id) REFERENCES users(id)
);
```

Açıklamalar:

- `user_id`, `users` tablosundaki `id` alanına referans veren bir foreign key’dir.  
- `amount`, sipariş tutarını temsil eden zorunlu bir alandır.  
- `created_at`, siparişin oluşturulma tarihini ve saatini tutar.  
- `fk_orders_users` ismindeki constraint, foreign key kuralını adlandırmaktadır.  

Bu yapı ile:

- Her sipariş bir kullanıcıya bağlıdır.  
- Sisteme kaydı olmayan bir kullanıcı için sipariş oluşturulması, veri tabanı düzeyinde engellenebilir.  

---

### 2.5.4 Many-to-Many İçin Ara Tablo Örneği

Bir öğrenci–ders ilişkisini (Many-to-Many) modellemek için genellikle üç tablo kullanılır:

- `students`  
- `courses`  
- `student_courses` (ara tablo)  

Örnek tanım:

```sql
CREATE TABLE students (
    id    INTEGER PRIMARY KEY,
    name  VARCHAR(100) NOT NULL
);

CREATE TABLE courses (
    id    INTEGER PRIMARY KEY,
    name  VARCHAR(100) NOT NULL
);

CREATE TABLE student_courses (
    student_id  INTEGER NOT NULL,
    course_id   INTEGER NOT NULL,

    PRIMARY KEY (student_id, course_id),

    CONSTRAINT fk_sc_students
        FOREIGN KEY (student_id) REFERENCES students(id),
    CONSTRAINT fk_sc_courses
        FOREIGN KEY (course_id) REFERENCES courses(id)
);
```

Burada:

- `student_courses` tablosu iki foreign key içerir.  
- `PRIMARY KEY (student_id, course_id)` ifadesi ile bu iki alan birlikte bileşik bir primary key oluşturur.  
- Aynı öğrenci–ders eşleşmesinin birden fazla kez eklenmesi böylece engellenmiş olur.  

---

### 2.5.5 İsimlendirme ve Tasarım Önerileri

Tablo ve sütun tasarlarken şu ilkeler yol gösterici olabilir:

- Tablo isimleri, temsil ettiği varlığı açıkça ifade etmelidir (`users`, `orders`, `students`).  
- Sütun isimleri, anlamlı ve tutarlı olmalıdır (`created_at`, `updated_at`, `is_active`).  
- Her tabloda bir primary key bulunmalıdır.  
- İlişkiler foreign key alanları üzerinden, mümkünse açık isimlendirilmiş constraint’lerle tanımlanmalıdır.  
- Bire birden fazla alanı içeren bileşik primary key kullanılacaksa, bu karar bilinçli şekilde verilmelidir.  

Bu bölümdeki `CREATE TABLE` örnekleri, veri tipleri ve constraint’ler ile önceki başlıklarda anlatılan kavramların, şema tasarımı seviyesinde bir araya getirilmiş halini göstermektedir.
